# ADR-007: 監査ログ設計

## ステータス
承認済み

## コンテキスト
福祉相談システムでは、誰がいつどのデータにアクセスしたかの追跡が法的に求められる。
MVP段階ではエラーログのみで、正常系の操作ログが未記録だった。

## 決定

### Cloud Loggingベースの監査ログ

Express ミドルウェアで `res.on('finish')` イベントを利用し、全APIリクエストの完了時にJSON構造化ログを出力する。Cloud Runのstdout→Cloud Loggingパイプラインを活用する。

### 記録項目

| フィールド | 内容 | 例 |
|-----------|------|-----|
| staffId | 操作者の職員ID（未認証時は"anonymous"） | "staff-123" |
| method | HTTPメソッド | "GET", "POST" |
| path | リクエストパス | "/api/cases/abc" |
| statusCode | レスポンスステータスコード | 200, 403 |
| durationMs | リクエスト処理時間（ミリ秒） | 45 |
| ip | クライアントIP | "203.0.113.1" |
| userAgent | User-Agentヘッダー | "Mozilla/5.0..." |

### 個人情報の非出力

- リクエストボディ（相談内容、個人名等を含む可能性）は**記録しない**
- パスパラメータ（ケースID等）のみ記録し、データの内容は記録しない
- レスポンスボディも記録しない

### 対象外

- `/health` エンドポイント（ヘルスチェックのノイズ除外）
- 静的ファイルリクエスト（フロントエンドアセット）

### 保持期間

Cloud Loggingのデフォルト保持期間（30日）を適用。
長期保持が必要な場合はCloud Loggingのシンク機能でCloud Storageにエクスポートする。

## 結果

- 全API操作が自動的に記録され、不正アクセスの追跡が可能になる
- Cloud Loggingのフィルタ機能で特定の職員・操作・時間帯の検索が可能
- 個人情報を記録しないため、ログ自体のセキュリティリスクを最小化
- パフォーマンス影響は最小限（非同期ログ出力、追加のI/Oなし）
